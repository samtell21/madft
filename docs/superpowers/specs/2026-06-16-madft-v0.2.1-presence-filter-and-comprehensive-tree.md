# madft v0.2.1 — presence filter + comprehensive default tree

Status: approved design (pending spec review). Builds on v0.2.0
(`2026-06-16-madft-v0.2-design.md`).

v0.2.1 ships a much larger built-in category tree and a **presence filter** that,
by default, hides anything the local machine can't act on. The filter is the
enabling mechanism: it lets the project ship a fat taxonomy (and, later, read an
even fatter one from a remote DB) while `ls`/`set` still show only what's useful
on *this* machine. No `--json` schema changes, so this stays a patch/minor bump to
**0.2.1**.

## 0. Scope

Two coupled deliverables:

1. **Presence filter** (code) — engine + CLI + tests.
2. **Comprehensive default tree** (data) — a rewritten `data/categories.toml`.

They ship together: the comprehensive tree is only pleasant to use *because* the
filter hides its inert long tail on any given system.

## 1. Presence filter

### 1.1 What "inert" means

- A **type** is *inert* when no installed app declares it — `applicable_count == 0`
  (i.e. `appindex.apps_for_type(t)` is empty). A type that has apps but no current
  default is **not** inert — those are the most interesting rows in a set-defaults
  tool, and must always show.
- A **category** is *inert* when its entire subtree contains no app-backed type
  (every type under it, recursively, is inert).

### 1.2 Default behavior and the flag

- Default view **hides** inert types and inert categories.
- A global `-a` / `--all` flag (declared like `--json`, `#[arg(short = 'a', long, global = true)]`)
  reveals the full taxonomy.
- The filter applies **uniformly to human and `--json`** output — the default view
  is the filtered view in both; `--all` reveals everything in both. Only *content*
  changes; the JSON schema is unchanged.

### 1.3 The explicit-mimetype exception

The filter drops inert types only from **multi-type views**: category/root umbrellas
and node listings. **Explicit selections are never filtered** — the user named them,
so they always win:
- A **single mimetype target** (`madft set jupyter text/x-ipynb`, `madft apps image/x-foo`)
  is always honored, even with `applicable_count == 0`.
- A **`set --types a,b` list** names exact types; those bypass the presence filter
  (so `set X Cat --types <inert> --force` can still force-set the named inert type).

Concretely: for `apps`/`set`, the presence filter applies only when the target is a
category/root AND (for `set`) no `--types` is given; a mimetype target or a `--types`
list skips it; `--all` skips it everywhere. `ls`/`types` always take a category/root,
so they always filter unless `--all`.

### 1.4 Per-command effect

- **`ls [PATH]`** — drop inert direct types; drop subcategories whose subtree is
  fully inert. Root `ls` drops fully-inert top-level categories.
- **`types <PATH>`** — drop inert types from the recursive list.
- **`apps [PATH|mimetype]`** — the reported `types` array reflects the filtered
  umbrella (a mimetype target is never filtered, per §1.3). The ranked app list is
  unaffected in practice (inert types contribute no apps).
- **`set <app> [PATH|mimetype]`** — the umbrella defaults to app-backed types, so
  `skipped_types` shrinks to the useful signal ("types other apps handle but this
  one doesn't"), and `set <app> <category> --force` only force-sets app-backed types
  unless `--all` is given. A single explicit mimetype target is never filtered.
- **`info` / `get` / `app` / `unset` / `init`** — no umbrella/listing; the flag is a
  no-op (an app's declared types are never inert — the app itself declares them).

The `set` empty-candidate guard is unchanged in spirit: if, after filtering and the
declare/force partition, no candidates remain, it still errors
`AppHandlesNothingUnderUmbrella`. (With the filter, this also naturally covers
"the whole umbrella is inert" — there were no app-backed types to set.)

### 1.5 Empty-after-filter UX

When a target resolves to a real node but everything filters out (no visible
subcategories and no visible types):

- **human**: print a hint, e.g. `(nothing installed under Media — use --all to see the full taxonomy)`.
- **`--json`**: return the normal structure with empty arrays (no special field).

### 1.6 Engine shape

- `ls`, `types`, `apps` gain a `show_all: bool` parameter.
- `set` currently takes four trailing booleans-ish (`force`, `no_clobber`,
  `dry_run`) plus the new `show_all`. Fold them into a small internal options
  struct to keep the signature readable:

  ```rust
  #[derive(Clone, Copy, Debug, Default)]
  pub struct SetOptions {
      pub force: bool,
      pub no_clobber: bool,
      pub show_all: bool,
      pub dry_run: bool,
  }
  // pub fn set(&self, app: &str, target: Option<&str>, types_filter: Option<&[String]>, opts: SetOptions) -> Result<SetPlan>
  ```

  This is an **internal** API change (engine + its tests); the CLI surface and the
  `--json` contract are unaffected.
- A private helper computes inert-ness:
  - `fn type_is_inert(&self, t: &MimeType) -> bool { self.appindex.apps_for_type(t).is_empty() }`
  - `fn subtree_has_app_backed_type(&self, id: CategoryId) -> bool` — any type in
    `tree.types_under(id)` that is not inert.
- The umbrella filter for `apps`/`set` (category/root targets only): retain
  app-backed types unless `show_all`.

### 1.7 CLI shape

- `Cli` gains `#[arg(short = 'a', long, global = true)] pub all: bool`.
- `execute`/`run_command` thread `show_all` alongside `json`.
- `ls`/`types`/`apps` dispatch passes `cli.all`; `set` builds a `SetOptions` from
  `*force`, `*no_clobber`, `cli.all`, `*dry_run`.

## 2. Comprehensive default tree

### 2.1 Placement principles

- **Earn the category.** A category exists if an app could plausibly be the default
  for everything under it. Don't group for grouping's sake. Deep nesting is fine
  when meaningful (`Text.Development.Python`).
- **Single placement.** Each type lives under exactly one path (enforced by the
  parser). Aliases canonicalize in `merge`, so listing an alias is harmless but the
  canonical form is preferred.
- **Quoted dotted keys** (`["Text.Development.Python"]`); ancestors auto-vivify, so
  only leaf paths are written.
- **Common desktop types, deep where it pays.** This tree targets the file types a
  desktop user realistically encounters (~180). The exhaustive long tail stays in
  `Other` today and is the job of the future remote DB; the presence filter makes a
  fat tree painless either way.

### 2.3 Notable placement decisions (single-placement forces one home)

- `svg → Images.Vector`; camera RAW → `Images.Raw`; `xcf → Images` (GIMP raster).
- `pdf/postscript/djvu/dvi/xps → Documents`; `csv/tsv → Documents.Office.Spreadsheet`;
  ebook formats → `Documents.Ebooks`.
- `json/yaml/toml/xml/ini/.desktop → Text.Development.Config`; `css/scss/less →
  Text.Development.Web`; per-language source → `Text.Development.<Lang>`. The Jupyter
  notebook type (`application/x-ipynb+json`) sits on `Text.Development` itself
  (language-agnostic), making it an interior node that owns a type *and* has
  subcategories — the first default-tree node to exercise v0.2.0's
  `categories:`/`types:` `ls` headers.
- `html/xhtml/feeds → Web`.
- `deb/rpm/flatpak/appimage/apk/snap → Archives.Packages`; `iso/dmg/raw →
  Archives.DiskImages`; compressed archives → `Archives`.

  reveals, explicit mimetype bypass).

## 4. Testing

Engine unit tests run against the committed `tests/fixtures` mime/app set (not the
shipped tree), so they stay deterministic. Add fixtures or use existing inert types:
- `type_is_inert` true for a fixture type with no declaring app, false for one with apps.
- `ls` hides an inert direct type and a fully-inert subcategory by default; `show_all` includes them.
- `types` drops inert entries unless `show_all`.
- `set <app> <category>` umbrella excludes inert types by default → `skipped_types`
  contains only app-backed undeclared types; `--all` restores the full umbrella
  (and `--all --force` force-sets inert types).
- explicit mimetype target with 0 apps is NOT filtered (`set`/`apps`/`info` still act on it).
- empty-after-filter `ls` prints the hint (human) / empty arrays (json).

Data test (`source.rs`): the shipped `DEFAULT_CATEGORIES` parses; a sampling of
placements resolve to the expected paths (e.g. `text/x-python` parses under
`Text.Development.Python`); no `DuplicatePlacement`.

## 5. Acceptance criteria

- `madft ls` on a normal desktop shows only categories with installed handlers;
  `madft ls --all` shows the full taxonomy. Same for `--json` (content differs, schema doesn't).
- A type with apps but no default still shows by default (it's actionable).
- `madft set mpv Media` reports a tight `skipped_types`; `madft set mpv Media --force`
  does not touch inert types; `--all --force` does.
- `madft set jupyter text/x-ipynb` works even if `text/x-ipynb` has zero apps
  (explicit mimetype bypass).
- The shipped tree parses, single-placement holds, sampled types resolve to their
  intended categories.
- `cargo test` + `cargo clippy --all-targets -- -D warnings` pass. No let-chains (MSRV 1.85).
- `Cargo.toml` is `0.2.1`; README documents `--all`.

## 6. Out of scope

- The remote/community category DB (still just the `Source` trait seam) — the
  comprehensive tree is the local stand-in.
- A dedicated "list unset" / "list inert" query — composable from `ls --all`/`--json`.
- Any `--json` schema change. Other deferred seams from prior specs are unchanged.
